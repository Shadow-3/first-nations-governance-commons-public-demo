# Repository Governance

Initial developer/custodian: Barayamal / Dean Foley  
Current status: public demo and fake-data sandbox preparation

## Governance Boundary

Code maintainers can decide repository infrastructure changes.

Community authority decides real governance, real data use, real eligibility, real voting, real publication, and real pilot approval.

These are not the same thing.

## Maintainer Responsibilities

- Keep the public repository fake-data only.
- Review contributions for privacy, data sovereignty, cultural safety, accessibility, and implementation value.
- Maintain the public demo and package metadata.
- Keep release notes and deployment reports current.
- Reject or remove unsafe data or misleading claims.

## Contribution Decisions

Maintainers may accept:

- Documentation improvements.
- Static demo UI improvements.
- Fake-data sandbox mapping.
- Decidim configuration guidance.
- Accessibility fixes.
- Test coverage and QA evidence.

Maintainers must reject or pause:

- Real community records.
- Real member lists.
- Cultural material.
- Real voting or eligibility data.
- Claims of real mandate.
- Changes that blur community authority and code maintainership.

## Adding Maintainers

Additional maintainers should be added only when:

- Their role is clear.
- Their access is necessary.
- They accept the fake-data and authority boundary.
- The initial maintainer approves the change.
- Any required governance/advisor review has occurred.
