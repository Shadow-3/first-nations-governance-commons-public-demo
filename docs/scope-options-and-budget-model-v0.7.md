# Scope Options And Budget Model v0.7

Project: First Nations Governance Commons  
Initial developer/custodian: Barayamal / Dean Foley  
Purpose: Help choose a realistic next build scope.

## Important Note

This document does not provide vendor quotes or final costs. It gives a scope model that can be priced by Barayamal, a Decidim provider, or a technical partner.

## Option 1: Static Public Demo Only

Use when:

- The goal is early advisor/funder conversation.
- No Decidim hosting is ready.
- The project needs low-risk feedback.

Includes:

- Static public demo.
- Advisor review materials.
- Funder brief.
- Fake sample data.
- Shareable ZIP.

Does not include:

- Real Decidim instance.
- User accounts.
- Admin workflows.
- Email notifications.
- Real participation.

Best next decision:

Run advisor review.

## Option 2: Decidim Fake-Data Sandbox

Use when:

- Advisors support the direction.
- Barayamal wants a true Decidim proof point.
- Funders/builders need to see the platform in the real tool.

Includes:

- Decidim organisation.
- Participatory process.
- Proposals.
- Meetings.
- Survey.
- Accountability.
- Mandate Record page.
- Evidence pack page.
- Fake-data labels.
- Basic admin training.

Does not include:

- Real community data.
- Member verification.
- SMS participation.
- Custom modules.
- Real pilot facilitation.

Best next decision:

Choose hosting/build path.

## Option 3: Controlled Community Pilot

Use only when:

- Community authority is confirmed.
- Governance Charter is approved.
- Data Sovereignty Charter is approved.
- Privacy notice is reviewed.
- Cultural safety rules are approved.
- Local admins/facilitators are ready.

Includes:

- Real process configuration.
- Participant information.
- Local admin setup.
- Offline participation workflow.
- Meetings/yarning circle recording.
- Proposal and survey collection.
- Prioritisation/support/vote method.
- Accountability tracker.
- Mandate Record.
- Evidence pack.

Does not include unless separately scoped:

- Custom SMS.
- Advanced member verification.
- GIS/Country mapping.
- Partner portal.
- CRM/member database sync.

Best next decision:

Only proceed after sandbox and no-go gates pass.

## Option 4: Managed Governance Cloud

Use when:

- Multiple communities or organisations want shared infrastructure.
- Barayamal wants an ongoing product/service model.
- Funding supports operations, training, hosting, and support.

Includes:

- Multi-community structure.
- Shared templates.
- Local admin training.
- Managed hosting/support.
- Steward network.
- Open commons documentation.
- Potential custom modules.

Best next decision:

Develop after one or more pilots prove the operating model.

## Budget Categories To Price

| Category | Static Demo | Sandbox | Controlled Pilot | Managed Cloud |
|---|---|---|---|---|
| Product/design | Low | Medium | Medium | Ongoing |
| Decidim technical setup | None | Medium | Medium | High/ongoing |
| Hosting/devops | Low | Medium | Medium/high | Ongoing |
| Governance design | Medium | Medium | High | Ongoing |
| Data/privacy review | Low | Medium | High | Ongoing |
| Community facilitation | None | Low | High | Ongoing |
| Training | Low | Medium | High | Ongoing |
| Custom modules | None | Avoid | Optional | Possible |
| Evaluation | Low | Medium | High | Ongoing |

## Recommended Funding Ask

Ask for funding in stages:

1. Advisor review and sandbox scoping.
2. Decidim fake-data sandbox.
3. Controlled pilot.
4. Shared commons and managed service model.

Do not ask funders to pay for a real pilot until the governance, data, privacy, cultural safety, and admin gates are clear.
