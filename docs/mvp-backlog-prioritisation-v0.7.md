# MVP Backlog Prioritisation v0.7

Project: First Nations Governance Commons  
Purpose: Prioritise what to build now, later, and not yet.

## Build Now

These are needed for the next credible step.

| Item | Why |
|---|---|
| Advisor review process | Tests legitimacy and safety before more build. |
| Decidim fake-data sandbox | Shows the model in the real platform. |
| Fake-data warning system | Prevents confusion with real consultation. |
| Proposal examples | Demonstrates participation flow. |
| Meeting/yarning examples | Demonstrates offline parity. |
| Survey examples | Demonstrates structured input. |
| Accountability examples | Shows follow-through. |
| Mandate Record page | Shows the core evidence product. |
| Evidence pack page | Shows funder/partner/advisor output. |
| Admin playbook | Reduces founder dependency. |

## Build Soon

These should follow sandbox review.

| Item | Why |
|---|---|
| Advisor synthesis report | Converts feedback into decisions. |
| Improved Decidim templates | Makes the sandbox reusable. |
| Basic admin training materials | Helps local stewards learn process management. |
| Privacy notice template | Needed before real pilot. |
| Partner response rules | Needed before external partners are invited. |
| Offline upload workflow | Needed to prove yarning circle input can count. |
| Hosting/security checklist | Needed before live use. |

## Build Later

Only build after the operating model is proven.

| Item | Why Not Yet |
|---|---|
| SMS participation | Valuable but custom; prove workflow first. |
| Advanced member verification | Important for real voting; too heavy for public demo. |
| Cultural permission rules engine | Important but needs protocol co-design. |
| Partner portal | Should not exist before partner boundaries are settled. |
| GIS/Country mapping | High sensitivity and complexity. |
| CRM/member sync | High privacy and governance risk. |
| Automated evidence export module | Manual evidence pack is enough for first sandbox. |

## Do Not Build Yet

| Item | Reason |
|---|---|
| Real voting process | Eligibility and authority not approved. |
| Real member database | Data sovereignty and privacy risk. |
| Cultural archive | Not appropriate without explicit community protocol. |
| Public real pilot | Governance gates not passed. |
| AI summarisation of real records | Needs explicit data and cultural approval. |

## MVP Definition

The MVP is not "all features."

The MVP is:

**A fake-data Decidim sandbox that shows one complete process from information to proposals, meetings, survey, prioritisation concept, Mandate Record, and accountability.**

## Success Test

The MVP succeeds if advisors can answer:

- Is this legitimate enough to keep developing?
- What must change before a real pilot?
- Can Decidim support the workflow?
- Is the Mandate Record useful?
- Can Barayamal explain the model clearly to funders/builders?
